**Authentication & Guard Architecture**

- **Authentication Strategy:** Passport JWT via a NestJS `JwtStrategy`.
  - Implementation: [backend/src/auth/strategies/jwt.strategy.ts](backend/src/auth/strategies/jwt.strategy.ts#L1)
  - Guard wrapper: `JwtAuthGuard` extends `AuthGuard('jwt')` ([backend/src/common/guards/jwt-auth.guard.ts](backend/src/common/guards/jwt-auth.guard.ts#L1)).

**1. Authentication Method**

- **Strategy used:** `JwtStrategy` (Passport JWT). The strategy extracts the token from the `Authorization: Bearer <token>` header and validates using the `JWT_SECRET` config value.
  - See: [backend/src/auth/strategies/jwt.strategy.ts](backend/src/auth/strategies/jwt.strategy.ts#L1)
- **Token payload claims:** the application signs tokens with the following claims (see `AuthService.login` and `JwtPayload`):
  - `sub` → mapped to `userId` inside `JwtStrategy.validate`
  - `email`
  - `role` (string, values defined in `Role` enum)
  - Evidence: token created in [backend/src/auth/auth.service.ts](backend/src/auth/auth.service.ts#L1) and validated in [backend/src/auth/strategies/jwt.strategy.ts](backend/src/auth/strategies/jwt.strategy.ts#L1).

**2. Guard Architecture**

- **Guard composition used on protected routes:**
  - `@UseGuards(JwtAuthGuard, RolesGuard)` — first authenticates the JWT and then enforces role checks.
  - Example usage in controller: [backend/src/agency/agency.controller.ts](backend/src/agency/agency.controller.ts#L1)
    - `@UseGuards(JwtAuthGuard, RolesGuard)`
    - `@Roles(Role.AGENCY)` or `@Roles(Role.ADMIN)`
- **Roles decorator & guard:**
  - `@Roles(...)` is a metadata decorator implemented in [backend/src/common/decorators/roles.decorator.ts](backend/src/common/decorators/roles.decorator.ts#L1).
  - `RolesGuard` reads that metadata and checks `request.user.role` against required roles: [backend/src/common/guards/roles.guard.ts](backend/src/common/guards/roles.guard.ts#L1).
  - `Role` enum values: [backend/src/common/enums/role.enum.ts](backend/src/common/enums/role.enum.ts#L1) (`ADMIN`, `AGENCY`).

- **Exact decorator combination examples (as used in repo):**
  - `@UseGuards(JwtAuthGuard, RolesGuard)` + `@Roles(Role.AGENCY)` — for agency-only endpoints (see `AgencyController#getMe`, `AgencyController#updateMe`).
  - `@UseGuards(JwtAuthGuard, RolesGuard)` + `@Roles(Role.ADMIN)` — for admin-only endpoints (see `AgencyController#getAdminStats`, `findAll`).

**3. Tenant Context Binding (agencyId)**

- **How tenant scoping is enforced:**
  - Tokens do NOT include an `agencyId` claim. Instead, the app binds tenant context at request handling time by using the authenticated `userId` (from `request.user.userId`).
  - Controllers read the authenticated principal (via `@Req() req`) and pass `req.user.userId` into service methods that resolve or validate the agency. Example:
    - Controller call: `this.agencyService.getAgencyByUserId(req.user.userId)` — [backend/src/agency/agency.controller.ts](backend/src/agency/agency.controller.ts#L1)
    - Service resolves agency by `userId`: `prisma.agency.findUnique({ where: { userId } })` — [backend/src/agency/agency.service.ts](backend/src/agency/agency.service.ts#L1)
  - Booking scoping uses either the vehicle's `agencyId` (on create) or verifies `booking.agency.userId === req.user.userId` before allowing status updates: see [backend/src/booking/booking.service.ts](backend/src/booking/booking.service.ts#L1).

- **Security properties of this approach:**
  - Pros: the server enforces ownership checks server-side (service layer verifies `userId` ↔ agency relationship), reducing trust in client-supplied tenant identifiers.
  - Notes: because `agencyId` is not embedded in the token, the service performs a DB lookup to bind tenant context from the authenticated `userId`.
